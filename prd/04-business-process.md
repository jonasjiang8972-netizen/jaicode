# AI 编程 Agent CLI 工具 — 业务流程设计 PRD

| 文档信息 | 内容 |
|---|---|
| 产品名称 | Jaicode |
| 文档版本 | V1.0 |
| 关联文档 | 《市场分析与产品定位PRD》《功能设计PRD》《架构设计PRD》《前端界面设计PRD》 |
| 定位 | 定义产品从获客到使用、反馈、（未来）付费的端到端业务流程，区分"本期落地流程"与"预留业务流程" |

---

## 1. 用户全生命周期业务流程总览

```
获客/发现 → 下载安装 → 首次使用(Onboarding) → 日常任务执行 → 遇到问题/反馈
     │                                                              │
     └──────────────────────► 意向收集(Waitlist) ◄───────────────────┘
                                     │
                          [预留·二期] 授权购买与激活
                                     │
                          [预留·二期] 订阅续费/团队管理
```

**本期（Phase 1）业务闭环范围**：获客 → 下载安装 → Onboarding → 日常使用 → 反馈 → 意向收集
**预留（Phase 2+）业务范围**：购买 → 激活 → 续费 → 换设备 → 团队管理 → 退款

## 2. 本期核心业务流程详解

### 2.1 获客与下载安装流程

```
渠道曝光                     用户决策                     安装
┌──────────────┐      ┌──────────────┐      ┌──────────────────┐
│ 官网/GitHub/   │─────▶│ 查看README/    │─────▶│ Mac: brew install  │
│ 社区文章/口碑    │      │ 对比竞品        │      │ Win: 下载exe直接运行│
└──────────────┘      └──────────────┘      └──────────────────┘
```

**关键节点说明：**
- 官网/GitHub 仓库 README 是本期唯一的"转化页面"，需清晰说明：与Cursor/Trae等竞品的差异化定位（本地优先、双语、终端原生）
- 不设置任何注册/登录门槛，降低试用摩擦——这是本期区别于多数SaaS竞品的核心策略

**流程指标：**
| 指标 | 说明 |
|---|---|
| 安装转化率 | 访问README到实际下载安装的比例 |
| 渠道来源分布 | 用于判断内测阶段应重点投入哪个获客渠道 |

---

### 2.2 首次使用（Onboarding）业务流程

```
首次启动 jaicode
      │
      ▼
自动检测系统语言 (zh/en)
      │
      ▼
展示欢迎信息 + 内测版本提示（无授权拦截）
      │
      ▼
用户执行第一条命令（plan/code/debug/ask 任一）
      │
      ▼
若为首次使用 LLM 相关命令，且未配置 API Key
      │
      ├── 已配置环境变量中的默认Key → 直接可用
      └── 未配置 → 提示引导：jaicode config --api-key <key>
```

**业务目标**：让用户在**5分钟内**完成"安装→跑通第一个任务"的完整体验，这是内测阶段留存的关键先行指标。

---

### 2.3 日常任务执行业务流程（四种模式）

以 **Code 模式** 为例（其余模式流程结构一致，权限范围不同，详见功能设计PRD）：

```
用户输入任务描述
      │
      ▼
CLI 读取项目上下文
      │
      ▼
调用 LLM 生成变更方案
      │
      ▼
展示 Diff，等待用户决策
      │
      ├── 确认(y) → 备份原文件 → 写入变更 → 展示结果 → [每10次任务]触发意向收集提示
      │
      └── 拒绝(N) → 任务取消，文件系统无变化
```

**业务规则**：
- 用户拒绝的任务不计入"意向收集提示"的触发计数（只统计成功应用的任务，避免频繁打扰）
- 任务执行全程本地留痕（日志），为后续"内测问题排查"业务流程提供依据

---

### 2.4 问题反馈与支持业务流程

```
用户遇到问题
      │
      ▼
CLI 本地日志自动记录（~/.jaicode/logs/）
      │
      ▼
用户通过以下渠道之一反馈：
      ├── GitHub Issue（技术类问题，附日志片段）
      ├── 社区群/论坛（使用类问题）
      └── Waitlist页面留言（商业化意向/合作咨询）
      │
      ▼
研发/运营团队分类处理
      ├── Bug → 进入研发排期
      ├── 功能建议 → 进入功能设计PRD待评审列表
      └── 商业化意向 → 转入"意向收集与早期用户运营流程"（见2.6）
```

**业务规则**：本期不提供专职客服渠道，问题反馈以社区化、异步处理为主，需在README中明确设置反馈渠道与预期响应时间（如"48小时内在GitHub Issue响应"）。

---

### 2.5 版本更新与发布业务流程

```
研发完成功能开发/修复
      │
      ▼
CI自动化构建（跨平台交叉编译 Mac/Windows 二进制）
      │
      ▼
内部测试通过
      │
      ▼
打Tag发布 → 更新 Homebrew Tap / 官网下载链接
      │
      ▼
发布说明（Changelog）双语同步发布
      │
      ▼
用户获取更新：
      ├── Mac (Homebrew): brew upgrade jaicode
      └── Windows: 官网重新下载覆盖安装（本期不做自动更新机制）
```

**业务规则**：本期不实现自动更新检测与推送机制（属于P2范围），版本更新依赖用户主动检查，需在CLI启动信息中附加"当前版本 vX.X.X"，并可选提示"访问官网查看是否有新版本"。

---

### 2.6 意向收集与早期用户运营业务流程

```
用户触发意向收集提示（CLI内低频展示 / 主动访问Waitlist页）
      │
      ▼
提交邮箱到 Waitlist
      │
      ▼
运营团队定期（如每周）导出名单
      │
      ▼
分类打标签：个人开发者 / 企业咨询 / 媒体合作
      │
      ▼
针对"企业咨询"标签 → 人工邮件/会议触达，收集私有化/离线授权具体需求
      │
      ▼
需求汇总 → 反哺《市场分析与产品定位PRD》和二期授权模块的优先级排序
```

**业务目标**：本期没有真实付费流程，意向收集与人工触达是验证"买断制License Key"与"离线激活"这两个差异化定位是否有真实付费意愿的**核心业务闭环**，其数据直接决定二期是否值得投入授权/支付模块开发。

## 3. 预留业务流程（二期，本期不实现）

> 以下流程仅作**业务设计预留**，用于指导架构接口设计（对应架构设计PRD中的 `ILicenseProvider` / `IBillingProvider`），本期不投入开发与运营资源。

### 3.1 授权购买与激活业务流程（预留）

```
用户在Web控制台选择套餐（个人版/团队版/企业版）
      │
      ▼
跳转支付页（Stripe / 国内发卡平台）
      │
      ▼
支付成功 → 系统生成授权码 → 邮件发送给用户
      │
      ▼
用户在CLI中输入授权码
      │
      ├── 在线激活：CLI → 云端License API → 绑定硬件指纹 → 返回加密令牌
      │
      └── 离线激活（企业版）：
              用户在断网设备运行 `jaicode machine-id` 获取机器码
                    │
                    ▼
              Web控制台粘贴机器码 → 生成离线证书文件
                    │
                    ▼
              用户下载证书文件 → 拷贝至断网设备 → CLI本地验签激活
```

### 3.2 订阅续费与到期提醒业务流程（预留）

```
系统定时任务扫描授权到期时间
      │
      ▼
到期前7天 → 触发提醒
      ├── CLI启动时展示到期提醒（双语）
      └── 邮件提醒（若用户留有邮箱）
      │
      ▼
用户续费 → 授权有效期延长 → CLI本地缓存令牌刷新
      │
      ▼
若到期未续费 → 核心功能自动锁定，仅保留Ask模式等基础能力
```

### 3.3 换设备与设备解绑业务流程（预留）

```
用户在Web控制台"设备管理"页 → 选择原设备 → 解绑
      │
      ▼
新设备CLI输入相同授权码 → 云端校验席位余量 → 自动绑定新设备指纹
```

### 3.4 团队席位分配业务流程（预留）

```
团队管理员在Web控制台购买N个席位
      │
      ▼
邀请成员（邮箱邀请）
      │
      ▼
成员激活个人账号 → 消耗1个席位
      │
      ▼
管理员可在后台查看席位使用情况、回收闲置席位
```

### 3.5 退款业务流程（预留）

```
用户在Waitlist/邮件提出退款申请
      │
      ▼
运营人工审核（是否符合退款政策，如"7天内未大量使用"）
      │
      ▼
通过 → 调用支付渠道退款API → 授权码失效 → 设备解绑
```

## 4. 跨职能协作流程（本期）

```
研发团队 ──产品缺陷/日志──▶ 反馈处理流程(2.4)
    ▲                              │
    │                        分类打标签
    │                              │
产品团队 ◀──功能建议/需求洞察────────┘
    │
    └──企业咨询意向──▶ 商务/运营团队 ──▶ 人工触达(2.6) ──▶ 反哺定位PRD
```

**协作规则**：本期团队规模小，建议每周一次跨职能同步会，同步 GitHub Issue 处理进度、Waitlist 意向名单质量、以及是否出现启动"二期授权/支付模块"的明确信号（如连续多个企业咨询提出相同的离线部署需求）。

## 5. 流程指标衡量点汇总

| 流程 | 关键指标 |
|---|---|
| 获客与安装 | 安装转化率、渠道分布 |
| Onboarding | 首次任务成功率、"5分钟跑通"达成率 |
| 日常使用 | 任务完成率、周活跃留存 |
| 反馈处理 | Issue平均响应时长、Bug修复周期 |
| 意向收集 | Waitlist提交量、企业咨询占比、触达转化率 |
| （预留）购买激活 | 试用转付费率、激活成功率 |
| （预留）续费 | 到期续费率、流失率 |

## 6. 风险与异常分支

| 风险场景 | 业务应对 |
|---|---|
| 意向收集数据量过少，无法判断付费意愿 | 延长内测周期，加强README中差异化定位的表达，扩大获客渠道测试 |
| 大量企业咨询集中在"离线部署"而非"买断制" | 及时反馈调整二期优先级，优先开发离线授权而非在线订阅 |
| 无自动更新机制导致用户滞留旧版本 | 本期通过Changelog+社区渠道主动通知重大更新，观察是否需要提前引入自动更新检测 |
| 反馈渠道分散（GitHub/社区/邮件）导致信息丢失 | 每周汇总一次跨渠道反馈，避免遗漏商业化关键信号 |

## 7. 待评审事项

1. "5分钟跑通首个任务"是否需要通过内置的交互式Onboarding向导来保障，还是仅靠文档？
2. 意向收集数据积累到什么量级/什么信号强度，才触发二期授权模块正式立项？需要与商务团队定义明确的启动阈值。
3. 版本更新的自动检测机制是否需要提前到本期实现，避免内测阶段用户体验因版本碎片化而下降？
